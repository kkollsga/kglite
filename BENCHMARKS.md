# KGLite benchmarks

Wall-to-wall time per topic (each topic sums several individual queries), lower is better, on one synthetic knowledge graph that **every engine loads from identical bytes**. Generated by `python benchmarks/benchmark.py` — see [Reproduce](#reproduce).

**Dataset:** 25,333 nodes · 266,310 edges (org/social graph: Person/Company/Project/Skill/City). Scale `medium`, seed-deterministic.

| Topic | **kglite** | DuckDB | igraph | Kùzu | NetworkX | rustworkx |
|---|---|---|---|---|---|---|
| Bulk load | 98.5ms | 123.9ms | 13.5ms | 534.8ms | 206.6ms | 9.9ms |
| Scan & lookup | 2.7ms | 1.7ms | 18µs | 4.0ms | 1.1ms | 248µs |
| Filter & aggregate | 2.3ms | 842µs | 2.6ms | 2.2ms | 2.9ms | 1.9ms |
| Traversal (1–3 hop, filtered, deep) | 10.5ms | 9.4ms | 1.8ms | 20.42s | 4.3ms | 5.1ms |
| Pathfinding & patterns | 200.2ms | 675µs\* | 98.7ms\* | 232.4ms | 65.4ms | 300.7ms\* |
| Graph algorithms | 13.0ms | 1.6ms\* | 1.7ms | 4.4ms\* | 11.2ms | 4.6ms |
| Mutations | 4.6ms | 1.3ms | 4.8ms | 83.7ms | 311µs | 63µs |
| **Total** | **331.7ms** | 139.4ms\* | 123.2ms\* | 21.29s\* | 291.8ms | 322.6ms\* |

\* partial — that engine skipped or doesn't support one query in the topic.

> Every engine runs the **same** queries on the **same** data — including the variable-length `[:KNOWS*1..3]` traversal. A large traversal number (e.g. Kùzu's) is that engine's own planning of that identical query on a dense, hub-heavy `KNOWS` subgraph, not a handicap we imposed; results are digest-checked equal across engines. Run it yourself (below).

### What's being compared

These aren't all the same kind of tool — read the table with that in mind:

- **kglite** — Cypher graph engine, in-memory
- **DuckDB** — SQL / relational
- **igraph** — C algo library
- **Kùzu** — Cypher graph DB, embedded
- **NetworkX** — pure-Python algo library
- **rustworkx** — Rust algo library

> The native algorithm libraries (NetworkX / rustworkx / igraph) are fast on pure graph algorithms but have **no query language, no property filtering at the engine, and no persistence** — they're in-process toolkits, not databases. kglite is a full Cypher engine with storage modes, persistence, and embeddings that stays competitive with them on raw traversal/algorithms while beating the other query engines (Kùzu, Neo4j) on most workloads.

### Scaling

The table above is a `medium` graph chosen so every engine can run (NetworkX/igraph hold the whole graph in RAM). kglite itself scales far past that: the bundled `graphgen` generator streams **millions of nodes in bounded memory**, and kglite's `mapped`/`disk` modes load and query larger-than-RAM graphs — see `benchmarks/competitive/largescale/`.

## Reproduce

```bash
pip install kglite kuzu networkx rustworkx igraph duckdb
python benchmarks/benchmark.py            # default: medium scale
python benchmarks/benchmark.py --scale large
```

Each engine runs the *same* queries (shared seed-derived parameters) and prints a result digest per topic, so the comparison reflects equal work. Versions used for this run:

_kglite 0.11.1, DuckDB 1.5.3, igraph 1.0.0, Kùzu 0.11.3, NetworkX 3.6.1, rustworkx 0.17.1_

Run on {'platform': 'macOS-26.3-arm64-arm-64bit-Mach-O', 'processor': 'arm', 'python': '3.14.3'}. Numbers are min wall-time over repeats; absolute times vary by hardware — the *ratios* are the point.
