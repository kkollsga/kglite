## Section 3 â€” `from_blueprint` deep-dive

**Question:** Does Phase 3 (dataset lifecycle lift) require lifting
`from_blueprint` itself, or is it already in core?

**Answer:** Already in core. **Phase 3 is not blocked here.**

### Evidence

Core `crates/kglite/src/graph/blueprint/mod.rs` re-exports:

```rust
pub mod build;
pub mod compute;
pub mod csv_loader;
pub mod csv_stream;
pub mod expr;
pub mod filter;
pub mod geometry;
pub mod schema;
pub mod timeseries;
pub mod validation;
pub use build::build;
pub use schema::load_blueprint_file;
```

Wheel `crates/kglite-py/src/graph/pyapi/blueprint.rs:23` defines
`from_blueprint_rust` (176 LOC) which:

1. Resolves the blueprint path (wheel ergonomics)
2. Calls `blueprint::load_blueprint_file(&bp_path)` (line 68) â€” core
3. Calls `blueprint::build(...)` â€” core
4. Applies `lock_schema` on the result â€” small helper
5. Returns the populated `KnowledgeGraph` PyClass

So the heavy lifting (CSV streaming, schema validation, FK resolution,
node/connection materialization, timeseries inlining, geometry
parsing) is ALL in core today. The wheel adds ergonomics + PyO3
marshalling.

### What Phase 3 needs

For SEC.open / Sodir.open / wikidata.open to lift to core, they need
to be able to call the build path. Two changes needed before Phase 3
implementation:

1. **Re-export `blueprint` through `api`** â€” 3 lines in
   `crates/kglite/src/lib.rs`:
   ```rust
   pub mod blueprint {
       pub use crate::graph::blueprint::{build, load_blueprint_file};
       pub use crate::graph::blueprint::schema::*; // if SchemaDefinition is needed
   }
   ```
2. **Audit `lock_schema`** â€” small Rust helper in wheel that may need
   to lift if datasets-in-core need it.

Estimated effort: **5-30 minutes** of api hygiene, no real lifting.

## Section 7 â€” Decisions for the binding-implementer guide (Phase 2)

These decisions emerged from the audit and should be documented in
`docs/guides/implementing-a-binding.md`:

1. **Bindings depend on `kglite::api::*` only.** Never reach into
   `kglite::graph::*`. If you need something not in api, file an
   issue â€” that's a missing re-export, not an excuse to bypass.
2. **The thin `kglite::api::KnowledgeGraph`** is the wrapper-type
   template. Each binding wraps `Arc<DirGraph>` + their own state
   (embedder cache, GIL handle, etc.).
3. **Cypher pipeline canonical entry point is `api::session::execute_read`
   / `execute_mut`.** Bindings should NOT roll their own
   `CypherExecutor`-driving loop; the session functions handle
   parsing, planning, schema validation, transactions, and CoW.
4. **Embedder registration** is via the `api::Embedder` trait. Bindings
   wrap their native embedder (ONNX, OpenAI, etc.) and pass an
   `Arc<dyn Embedder>` to the graph. The `FastEmbedAdapter` shows
   the pattern.
5. **Datasets:** after Phase 3, `api::datasets::sec::open`,
   `api::datasets::sodir::open`, `api::datasets::wikidata::open`
   return `Arc<DirGraph>` directly. Until then, bindings must
   re-implement the lifecycle orchestration (~1356 LOC of Python
   logic today).
6. **Error handling:** all api functions return `Result<T, KgError>`.
   `KgError` carries a stable `KgErrorCode`. Bindings should map
   `KgErrorCode` to their language's idiomatic error types.
7. **Out of scope for v1 bindings:** NumPy/Pandas/Arrow ingestion
   helpers, Python callback support, fluent-API selection chains.
   These are language-specific ergonomics layers that each binding
   builds on top of the api.

## Section 8 â€” What I'm NOT recommending

To avoid scope creep:

- **No grand `api` restructure.** The re-export pattern at
  `lib.rs:53-109` works. Don't move it into a dedicated `api/`
  directory unless it grows past ~100 items.
- **No `#[deprecated]` on internal modules.** They're `pub` for
  in-crate reasons (separate-file organization); they're not part
  of the stability contract. Documenting "depend on `api`" is
  enough.
- **No semver enforcement tool yet.** Cargo's `package.publish`
  + manual changelog discipline are sufficient at 0.10.x scale.
  Revisit at 1.0.
- **No breaking changes to the wheel's PyClass surface.** This audit
  is about Rust core's stable surface; the wheel's Python API is its
  own concern.
